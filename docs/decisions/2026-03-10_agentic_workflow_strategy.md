# Architecture Decision Record (ADR): Agentic Workflow Strategy
**Date:** March 10, 2026

## Context & Motivation
The project has evolved from a simple Python automation script into a multi-agent "Venture Analyst" pipeline. To scale this effectively without turning the codebase into unmaintainable "AI spaghetti code" (Tech Debt), we need a rigid, professional framework for how AI agents interact with the code and perform tasks.

This strategy is synthesized from three major industry benchmarks discussed on this date:
1.  **Joshua Morony's "Phalanx" Workflow:** The human acts as the Manager/Architect, writing `plan.md` files. AI agents act as "Workers", executing these isolated plans in separate branches/worktrees.
2.  **Stripe's "Minions":** Autonomous coding agents that run in isolated Cloud Devboxes, communicate via Slack/Telegram, heavily rely on MCP (Model Context Protocol) for internal knowledge, and are forced through strict CI/CD "Quality Gates" (linters/tests) before merging.
3.  **Andrej Karpathy's "Autoresearch":** An autonomous AI research sandbox that enforces extreme restrictions: specific files are designated "Read-Only Infrastructure" (to prevent the AI from cheating metrics), while only a single target file is exposed for experimentation.

## Key Decisions & Architectural Rules

### 1. The Manager-Worker Divide
- **Humans:** Do not write boilerplate. We write architecture, interfaces, and strict markdown instructions (`plan.md` or `program.md`).
- **Agents:** Never push directly to `main`. They act on isolated branches based on the markdown instructions.

### 2. Read-Only Infrastructure (The Karpathy Sandbox)
To prevent agents from manipulating their own success metrics or breaking core systems (like API key rotation or Playwright logic):
- **Infrastructure Code:** Core orchestration, evaluation metrics, and API handling are **READ-ONLY** for generic workers.
- **Playground Code:** Only specific files/modules are exposed for the agent to modify during a given task.

### 3. Shift-Left Quality Gates (The Stripe Model)
An agent's Pull Request is invalid unless it passes automated checks.
- Code generated by agents must pass local linters (e.g., Ruff) and basic unit tests before human review. This is an automated loop; the agent must fix its own syntax errors.

### 4. The Simplicity Criterion
All else being equal, simpler code is better. If an agent achieves a marginal performance gain but introduces massive complexity, the PR is rejected. Code reduction that maintains functionality is considered a massive success.

### 5. Multi-Agent Cloud Orchestration (Future Roadmap)
- We will migrate away from local Windows Docker execution to prevent blocking the host machine.
- **Target Architecture:** A cheap Cloud VPS (e.g., Hetzner) running Docker containers.
- **Trigger:** A Telegram Bot interface where the user texts `/research [topic]`.
- **Workflow:** The cloud agent clones the repo, creates a branch, runs the pipeline, and submits a PR to GitHub asynchronously.

## Conclusion
This document serves as the foundational "Context Seed" for all future AI agents interacting with this repository. If you are an AI reading this, adhere strictly to the Manager-Worker isolation, respect Read-Only boundaries, and prioritize simplicity.
